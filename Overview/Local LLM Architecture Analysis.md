# Architecture Analysis Pipeline

The `LocalLLMAnalysis/` directory contains a multi-pass documentation pipeline that generates architecture documentation for game engine codebases. It uses a local Ollama LLM server for analysis and optionally clangd for semantic symbol extraction. All scripts run from the repository root.

---

## How the Pipeline Works

The pipeline reads source files, sends them to a local LLM one at a time, and builds up layers of documentation -- starting with per-file descriptions, then cross-references, diagrams, and a synthesized overview. Each layer feeds into the next.

Output goes to `architecture/` at the repo root. Configuration lives in `LocalLLMAnalysis/.env`.

---

## Pipeline Steps

The six steps run in order. Steps marked **free** make zero LLM calls and finish in seconds.

```
Step 0 (free)    serena_extract          LSP symbol extraction (optional)
Step 1           archgen_local           Per-file architecture docs
Step 2 (free)    archxref                Cross-reference index
Step 3 (free)    archgraph               Call graph diagrams
Step 4           arch_overview_local     Subsystem architecture overview
Step 4b (free)   archpass2_context       Targeted context per file
Step 5           archpass2_local         Selective re-analysis with context
```

### Step 0 -- LSP Symbol Extraction (optional)

**Scripts:** `serena_extract.ps1` + `serena_extract.py`

Extracts ground-truth symbol definitions and cross-file references from clangd (C++ language server). This step is entirely optional -- the pipeline works without it, but when present, the symbol data enriches the per-file docs in Step 1.

Requires `compile_commands.json`, which is generated by `generate_compile_commands.py`. That script auto-detects the build system (`.vcxproj`, `.vcproj`, `.dsp`, CMake, Meson, Ninja, or Bazel) and produces the compilation database clangd needs.

The Python script (`serena_extract.py`) spawns clangd processes in parallel, manages a shared work queue, monitors RAM, and handles crash recovery. clangd builds its index on-the-fly during the first run (subsequent runs are faster thanks to the cached index at `.cache/clangd/index/`).

Output: `LocalLLMAnalysis/.serena_context/` -- one `.serena_context.txt` file per source file.

### Step 1 -- Per-File Architecture Docs

**Script:** `archgen_local.ps1`

The foundation of the pipeline. For each source file matching the configured preset, the script:

1. Reads the source (truncated to `MAX_FILE_LINES` with head+tail if too long)
2. Loads compressed LSP symbols if Step 0 was run
3. Sends both to the LLM with a prompt asking for purpose, responsibilities, key types, key functions, globals, and dependencies
4. Writes the response as a `.md` file mirroring the source tree layout

Processing is single-threaded (one GPU inference at a time). An SHA1 hash database tracks which files are done, so the script can be interrupted and resumed. Trivial or generated files are detected and given stub docs without an LLM call.

Output: `architecture/<relative_path>.md` for each source file.

### Step 2 -- Cross-Reference Index

**Script:** `archxref.ps1`

Parses the per-file docs from Step 1 and builds a cross-reference index: function-to-file mappings, call graph edges, global state ownership, header dependencies, and subsystem interfaces. Pure text processing -- no LLM calls.

Output: `architecture/xref_index.md`

### Step 3 -- Call Graph Diagrams

**Script:** `archgraph.ps1`

Generates Mermaid diagrams from the Step 1 docs: function-level call graphs grouped by subsystem and subsystem-level dependency diagrams. Pure text processing -- no LLM calls.

Output: `architecture/callgraph.md`, `architecture/callgraph.mermaid`, `architecture/subsystems.mermaid`

### Step 4 -- Architecture Overview

**Script:** `arch_overview_local.ps1`

Synthesizes all the per-file docs into a subsystem-level architecture overview. For large codebases, it automatically chunks the input (threshold: 400 lines of extracted headings) and produces per-subsystem overviews first, then a final synthesis combining them all.

To stay within the LLM's context window, only the heading and purpose line from each per-file doc are sent -- not the full content.

Output: `architecture/architecture.md` (plus per-subsystem files in chunked mode)

### Step 4b -- Targeted Pass 2 Context

**Script:** `archpass2_context.ps1`

Prepares per-file context bundles for Step 5 by extracting the relevant architecture overview paragraphs and cross-reference entries for each file. This narrows ~500+ generic context lines down to ~30-80 targeted lines per file, keeping Step 5 within the LLM's token budget. No LLM calls.

Output: `architecture/.pass2_context/<path>.ctx.txt`

### Step 5 -- Selective Re-Analysis

**Script:** `archpass2_local.ps1`

Re-analyzes selected files with the architecture context gathered in Steps 2 and 4. Each file gets its source (capped at 300 lines), its Step 1 doc, and its targeted context from Step 4b. The LLM enriches the doc with architectural role, cross-references, and design patterns.

Files can be selected by score (`-Top N`), by path (`-Only`), or scored without running (`-ScoreOnly`). Like Step 1, processing is SHA1-cached and resumable.

Output: `architecture/<relative_path>.pass2.md`

## Orchestration

**Script:** `Arch_Analysis_Pipeline.py`

Automates running all six steps across multiple subsections of a codebase. It reads the `#Subsections begin` / `#Subsections end` block from `.env` (e.g. `REDALERT`, `TIBERIANDAWN`, `CnCTDRAMapEditor`), then for each subsection runs Steps 1 through 5 in order. After each subsection completes, the `architecture/` folder is renamed to `N. <subsection>/`.

Subsections that already have a numbered output folder are skipped, making the pipeline resumable at the subsection level.

Options: `--dry-run` (simulate), `--start-from N` (skip subsections), `--skip-lsp` (omit Steps 0).

---

## Shared Infrastructure

**Script:** `llm_common.ps1`

Dot-sourced by every `*_local.ps1` script. Provides the LLM client (`Invoke-LocalLLM` with retry logic), `.env` parsing, SHA1 hashing, preset definitions, trivial file detection, source truncation, output budgeting, LSP context loading, and progress display.

---

## Configuration

All configuration lives in `LocalLLMAnalysis/.env`. Key settings:

| Setting              | Purpose                                             |
|----------------------|-----------------------------------------------------|
| `LLM_HOST` / `PORT`  | Ollama server address                               |
| `LLM_MODEL`          | Model name (e.g. `qwen2.5-coder:14b`)              |
| `PRESET`             | Codebase type (`cnc`, `quake`, `unreal`, `godot`, etc.) |
| `MAX_FILE_LINES`     | Source truncation limit (default 800)               |
| `SKIP_TRIVIAL`       | Skip generated/trivial files (default on)           |
| `INCLUDE_EXT_REGEX`  | Override file extensions to include                 |
| `EXCLUDE_DIRS_REGEX` | Override directories to exclude                     |

## Key Design Decisions

- **Single-threaded LLM calls.** The local GPU handles one inference at a time, so parallelism is at the script level (text-processing steps run fast between LLM steps), not within LLM calls.
- **Aggressive context management.** Source is truncated, LSP data is compressed, output budgets are adaptive, and overview synthesis uses extracted headings only -- all to fit within the local LLM's ~32K token context window.
- **Full resumability.** Every LLM-calling script uses SHA1 hashing to skip files that haven't changed. Interrupt with Ctrl+C and re-run to continue where you left off.
- **No LLM lock-in.** The pipeline talks to any Ollama-hosted model via the OpenAI-compatible API. Swap models by changing `LLM_MODEL` in `.env`.
