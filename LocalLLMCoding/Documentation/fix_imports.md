# fix_imports.py

## Purpose

`fix_imports.py` is a post-generation repair tool that detects and fixes cross-file import errors in a Python package. After `run_aider.py` generates all source files for a project, each file may contain import statements that reference symbols with slightly wrong names, missing attributes, or non-existent modules -- a common drift mode when each file is generated in its own isolated aider session without full visibility into what other steps actually produced.

The tool works in iterative passes: it walks every `.py` file under a package directory, attempts to import each module in a subprocess, collects failures with their full tracebacks, identifies the related "other file" involved in each cross-file mismatch, deduplicates failures that share the same edit scope, and invokes aider to reconcile the mismatches. After each repair pass, it re-checks all modules. The loop continues until all imports succeed or the maximum iteration count is reached.

## Prerequisites

### Tools

- **Python 3.10+** with the target project's virtual environment activated. The package must be importable (typically via `pip install -e .` in the project root).
- **aider** CLI installed and on PATH (`pip install aider-chat`).
- **Ollama** server running and accessible at the configured endpoint.

### Configuration

- `Common/.env` must exist with Ollama endpoint configuration (`LLM_HOST` + `LLM_PORT` or `LLM_ENDPOINT`) and optionally `LLM_FIX_IMPORTS_MODEL`.

### Prior Pipeline Steps

- The target package's source files must already exist (generated by `run_aider.py` or written manually).
- The package must be installed in editable mode (`pip install -e .`) so that `import pkg.module` works from the Python interpreter.

## Usage

```
python fix_imports.py [options]
```

Run from the **target project's root directory** with the project's virtual environment activated.

### CLI Options

| Parameter | Type | Default | Description |
|---|---|---|---|
| `--package DIR` | str | `src/nmon` | Package directory to walk. All `.py` files under this directory are discovered and import-tested. Path is resolved relative to the current working directory. |
| `--python EXE` | str | current interpreter | Python executable to use for import checks. Should point to the project venv's Python so that all dependencies are available. Defaults to `sys.executable` (the interpreter running the script). |
| `--max-iters N` | int | `3` | Maximum number of fix-check-fix iterations before giving up. Each iteration re-scans all modules, so setting this higher allows more repair passes but takes longer. |
| `--log PATH` | str | `fix_imports.log` | Path to the log file that records each iteration's failures, error details, and aider invocations. Relative paths are resolved from the current working directory. Parent directories are created if needed. |

## How It Is Invoked

### Standalone

```bash
cd C:\Coding\myproject
python C:\Coding\LocalLLM_Pipeline\LocalLLMCoding\fix_imports.py --package src/myapp
```

### Called by ArchPipeline.py (Coding Mode)

When the coding mode is fully wired, `fix_imports.py` will be invoked as Stage 5 (import repair) after `run_aider.py` completes Stage 4 (code generation).

### Typical Pipeline Sequence

1. Planning stages generate `aidercommands.md`
2. `run_aider.py` generates all source files (Stage 4)
3. `fix_imports.py` repairs import mismatches (Stage 5)

## Input Files

| File | Description |
|---|---|
| `Common/.env` | Shared configuration: Ollama endpoint and fix-imports model name. |
| All `.py` files under `--package` | Discovered via recursive glob. Each is tested by attempting `import <dotted.module.name>` in a subprocess. |

### Module Discovery

The tool converts file paths to dotted module names relative to the package directory's parent. For example, if `--package src/nmon` is set:

- `src/nmon/config.py` becomes `nmon.config`
- `src/nmon/ui/__init__.py` becomes `nmon.ui`
- `src/nmon/ui/dashboard.py` becomes `nmon.ui.dashboard`

Files named `__init__.py` are imported as their parent package (the `__init__` part is dropped).

## Output Files / Directories

| Output | Description |
|---|---|
| Modified `.py` files in the package | Aider edits files in-place to fix import mismatches. |
| `fix_imports.log` (or `--log` path) | Detailed log of every iteration: timestamps, failing module summaries, full error tracebacks, related files identified, and aider invocations. |
| `.aider.*` files | Aider's own working files created in the working directory during repair sessions. |

## Environment Variables / .env Keys

| Key | Source | Description |
|---|---|---|
| `LLM_FIX_IMPORTS_MODEL` | `Common/.env` | Ollama model tag for the import-fix aider calls. Default if missing: `qwen3-coder:30b`. Kept separate from `LLM_AIDER_MODEL` so the repair pass can use a stronger model without affecting the main generation flow. |
| `LLM_HOST` | `Common/.env` | Ollama server hostname. |

Same as `run_aider.py` (handled by the shared `_pipeline.config.resolve_ollama_endpoint`):

1. `OLLAMA_API_BASE` environment variable
2. `LLM_ENDPOINT` environment variable
3. `LLM_ENDPOINT` from `.env`
4. `LLM_HOST` + `LLM_PORT` from `.env`
5. Hardcoded default: `http://192.168.1.126:11434`

## Key Behaviors

### Import Testing

Each module is tested by spawning a fresh Python subprocess that runs `import <module_name>`. This ensures a clean import environment with no cross-contamination between tests. The subprocess uses the interpreter specified by `--python` (default: the current one), so the project's venv dependencies are available.

The test output is captured: return code 0 means success, non-zero means the import raised an exception. The full stderr is captured for error analysis.

### Related File Detection

When an import fails, the tool parses the error traceback to identify the "other file" involved in the mismatch. Three regex patterns are tried:

1. `cannot import name 'X' from 'pkg.mod' (path)` -- extracts `pkg.mod`
2. `No module named 'pkg.mod'` -- extracts `pkg.mod`
3. `from pkg.mod import X` in traceback lines -- extracts `pkg.mod`

The dotted name is converted to a file path and checked for existence (both as `pkg/mod.py` and `pkg/mod/__init__.py`). When found, this related file is added to the aider edit scope alongside the failing module.

### Failure Deduplication

Multiple modules can fail due to the same root cause. For example, if `nmon.ui.main_window` has a broken import from `nmon.ui.sidebar`, then both `nmon.ui.main_window` and `nmon.ui.__main__` (which imports `main_window`) will fail with errors pointing to the same pair of files.

The tool groups failures by their **edit scope** -- the sorted set of file paths that aider would be asked to edit (target file + related file). Failures with identical scopes are combined into a single aider invocation with all their error messages concatenated. This prevents later aider calls from overwriting fixes made by earlier calls to the same files.

### Aider Invocation

Each unique edit scope triggers one aider call:

```
aider --no-git --yes --model <model> --message <combined_errors> <file1> [file2]
```

Key flags:
- `--no-git`: Aider does not create commits (the pipeline manages git separately).
- `--yes`: Auto-confirm all aider prompts (non-interactive).
- The message instructs aider to resolve the import error by aligning symbol names between files, without inventing new symbols, and preserving symbols used by other callers.

### Iterative Repair Loop

After all aider calls in an iteration complete, the tool re-checks every module from scratch. If failures remain, a new iteration begins. This handles cascading fixes: repairing file A may fix the import in file B, or may reveal a new error in file C that was previously masked.

The loop terminates when:
- All modules import cleanly (exit code 0), or
- `--max-iters` iterations are exhausted (exit code 1).

### Logging

The log file is written progressively during execution. Each iteration records:
- A summary section listing all failing modules with their one-line error
- A full-errors section with complete tracebacks and identified related files
- Timestamps for each iteration

The summary is written **before** aider calls begin, so if the process is interrupted, the log still contains the full diagnostic picture.

## Exit Codes

| Code | Meaning |
|---|---|
| `0` | All modules import cleanly (possibly after one or more repair iterations). |
| `1` | Still failing after `--max-iters` iterations. Manual intervention needed. |
| `1` (via `sys.exit`) | Package directory not found. |

## Examples

### Example 1: Default run against src/nmon

```bash
cd C:\Coding\nmonClaude
python C:\Coding\LocalLLM_Pipeline\LocalLLMCoding\fix_imports.py
```

Walks `src/nmon`, tests every `.py` module, and runs up to 3 repair iterations using the model from `LLM_FIX_IMPORTS_MODEL` in `Common/.env`.

### Example 2: Custom package with more iterations

```bash
python C:\Coding\LocalLLM_Pipeline\LocalLLMCoding\fix_imports.py \
  --package src/myapp \
  --max-iters 5
```

Targets a different package (`src/myapp`) and allows up to 5 repair passes. Useful for larger packages with deeply cascading import chains.

### Example 3: Use a specific Python interpreter and custom log path

```bash
python C:\Coding\LocalLLM_Pipeline\LocalLLMCoding\fix_imports.py \
  --package src/nmon \
  --python C:\Coding\nmonClaude\.venv\Scripts\python.exe \
  --log logs/import_fixes.log
```

Uses an explicit venv interpreter (useful when running the script from outside the project's venv) and writes the log to a custom location.

### Example 4: Single iteration for quick check

```bash
python C:\Coding\LocalLLM_Pipeline\LocalLLMCoding\fix_imports.py \
  --package src/nmon \
  --max-iters 1
```

Runs one pass only: tests all imports, invokes aider for any failures, then exits regardless of whether the fixes worked. Useful for a quick first pass when you plan to inspect results manually.

### Example 5: Minimal run to just identify failures (no aider)

There is no `--dry-run` flag, but you can achieve a diagnostic-only run by setting `--max-iters 1` and interrupting after the first scan prints results (before aider calls begin). Alternatively, inspect the log file which records all failures before invoking aider.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| "Package directory not found" | Wrong `--package` path | Verify the path relative to cwd. Use `--package src/myapp` not `--package myapp`. |
| All modules show FAIL with ModuleNotFoundError | Package not installed in editable mode | Run `pip install -e .` in the project root first. |
| Aider fixes one file but breaks another | Cascading dependencies | Increase `--max-iters` to allow more repair passes. |
| Same error persists across iterations | Model unable to resolve the mismatch | Check the log, fix manually, then re-run with `--max-iters 1` to verify. |
| Import test passes locally but fails here | Different Python interpreter | Use `--python` to point at the correct venv. |
