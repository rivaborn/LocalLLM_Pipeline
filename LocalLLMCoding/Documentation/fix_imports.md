# fix_imports.py

## Purpose

`fix_imports.py` is an **advisory import checker**. Walks every `.py` file under a package root, tries to import each module in a fresh Python subprocess, and ‚Äî for every failure ‚Äî asks the local LLM to diagnose the root cause and propose a specific fix.

**Does NOT apply any fixes.** The LLM's proposals are written to a log file for the user to implement manually (e.g. installing a missing package, editing a source file, invoking aider interactively, etc.).

### Design rationale

An earlier version of this tool invoked aider in a repair loop. That approach was unreliable: aider sometimes recommended installing a missing package and then launched the next iteration before the install completed, or applied a fix that masked a deeper drift. The current version stops at "here's what to do" so humans decide how to act on each proposal ‚Äî which matches the broader pipeline principle that automated systems should surface intent rather than silently edit code the user hasn't reviewed.

## Prerequisites

### Tools

- **Python 3.10+** with the target project's virtual environment activated. The package must be importable (typically via `pip install -e .` in the project root).
- **Ollama** server running and accessible at the configured endpoint.

No aider installation needed ‚Äî the advisory path uses only `invoke_local_llm` against the local Ollama server.

### Configuration

- `Common/.env` must exist with Ollama endpoint configuration (`LLM_HOST` + `LLM_PORT` or `LLM_ENDPOINT`). The model is resolved via `cfg.resolve_model(env, 'LLM_AIDER_MODEL', 'qwen3-coder:30b')` ‚Äî shares `LLM_AIDER_MODEL` with Stage 4. Set that key or `LLM_DEFAULT_MODEL` to override.

### Prior Pipeline Steps

- The target package's source files must already exist (generated by `run_aider.py` in Stage 4 of the coding pipeline, or written manually).
- The package must be installed in editable mode (`pip install -e .`) so that `import <pkg>.<module>` resolves cleanly from the Python interpreter.

## Usage

```
python fix_imports.py [options]
```

Run from the **target project's root directory** with the project's virtual environment activated.

### CLI Options

| Parameter       | Type | Default             | Description                                                                                                                                                                                                    |
| --------------- | ---- | ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--package DIR` | str  | `src/nmon`          | Package directory to walk. All `.py` files under this directory are discovered and import-tested. Path is resolved relative to the current working directory.                                                  |
| `--python EXE`  | str  | current interpreter | Python executable to use for import checks. Should point to the project venv's Python so all dependencies are available. Defaults to `sys.executable` (the interpreter running the script).                    |
| `--log PATH`    | str  | `fix_imports.log`   | Path to the log file that records failures, stderr tracebacks, related-file detection results, and (when diagnosis is enabled) the LLM's advisory per-failure. Parent directories are created if they don't exist. |
| `--no-diagnose` | flag | `False`             | Skip the LLM diagnosis step ‚Äî only list failures with their tracebacks. Useful when you want the bare failure list quickly without waiting for a 30B model to generate explanations.                           |

(There is intentionally no `--max-iters`, no `--model` override, and no `--dry-run` ‚Äî the tool is single-pass by design.)

## How It Is Invoked

### Standalone

```bash
cd C:\Coding\myproject
python C:\Coding\LocalLLM_Pipeline\LocalLLMCoding\fix_imports.py --package src/myapp
```

### Called by ArchPipeline.py (Coding Mode)

`ArchPipeline.py coding` invokes `fix_imports.py` as **Stage 5** after `run_aider.py` completes Stage 4. Dispatch happens in `Common/_pipeline/modes/coding/stages_exec.py::stage5_fix_imports`, which runs the script as a subprocess. The pipeline passes `--package <auto-detected-or-explicit>` based on `--package-name` / `--package-dir` CLI flags or autodetection via `fileops.detect_package_dir()`.

### Typical Pipeline Sequence

1. Planning stages (0‚Äì3) generate `aidercommands.md`
2. Optional review stages (2c, 3c) audit + patch
3. `run_aider.py` generates all source files (Stage 4)
4. `fix_imports.py` checks imports and produces an advisory log (Stage 5)
5. User reads the log and applies suggested fixes manually if any failures remain

## Input Files

| File                              | Description                                                                                                |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `Common/.env`                     | Shared configuration: Ollama endpoint. Model resolved via `LLM_AIDER_MODEL` ‚Üí `LLM_DEFAULT_MODEL` ‚Üí `qwen3-coder:30b`. |
| All `.py` files under `--package` | Discovered via recursive glob. Each is tested by attempting `import <dotted.module.name>` in a subprocess. |

### Module Discovery

The tool converts file paths to dotted module names relative to the package directory's parent. For example, with `--package src/nmon`:

- `src/nmon/config.py` ‚Üí `nmon.config`
- `src/nmon/ui/__init__.py` ‚Üí `nmon.ui`
- `src/nmon/ui/dashboard.py` ‚Üí `nmon.ui.dashboard`

Files named `__init__.py` are imported as their parent package (the `__init__` part is dropped).

## Output Files / Directories

| Output                              | Description                                                                                                                                                             |
| ----------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `fix_imports.log` (or `--log` path) | Advisory report: header (timestamp, package, model, endpoint), per-failure summary, full stderr tracebacks, identified related-files, and per-failure LLM diagnoses. Written progressively during execution so partial results are preserved on interrupt. |

**No source files are modified by this tool** ‚Äî that's the key design change from the old version.

| `LLM_AIDER_MODEL`       | `Common/.env`  | Ollama model tag for the diagnosis LLM (shared with Stage 4 aider). Resolved via `cfg.resolve_model`: `LLM_AIDER_MODEL` ‚Üí `LLM_DEFAULT_MODEL` ‚Üí `qwen3-coder:30b`. |
| `LLM_HOST`              | `Common/.env`  | Ollama server hostname.                                                                                                                              |

| `OLLAMA_API_BASE`       | set at runtime | The resolved endpoint is written to this env var at startup so any child process (typically none in this tool) inherits the correct Ollama URL.      |

### Endpoint Resolution Precedence

Handled by the shared `_pipeline.config.resolve_ollama_endpoint` (same precedence as all other pipeline stages):

1. `OLLAMA_API_BASE` environment variable
2. `LLM_ENDPOINT` environment variable
3. `LLM_ENDPOINT` from `.env`
4. `LLM_HOST` + `LLM_PORT` from `.env`
5. Hardcoded default: `http://192.168.1.126:11434`

## Key Behaviors

### Import Testing

Each module is tested by spawning a fresh Python subprocess that runs `import <module_name>`. Clean import environment with no cross-contamination between tests. The subprocess uses the interpreter specified by `--python` (default: the current one) so the project's venv dependencies are available.

Test outcome: return code 0 means success; non-zero means the import raised. Full stderr is captured for error analysis.

### Related-File Detection

When an import fails, the tool parses the error traceback with three regex patterns to identify the other file involved in the mismatch (typically the symbol's canonical source):

1. `cannot import name 'X' from 'pkg.mod' (path)` ‚Üí extracts `pkg.mod`
2. `No module named 'pkg.mod'` ‚Üí extracts `pkg.mod`
3. `from pkg.mod import X` in traceback lines ‚Üí extracts `pkg.mod`

The dotted name is converted to a file path and checked for existence (both as `pkg/mod.py` and `pkg/mod/__init__.py`). When found, the related file is logged alongside the failing module and fed into the LLM's diagnosis prompt so it sees both sides of the mismatch.

### LLM Diagnosis

For each failure (unless `--no-diagnose` is set), the tool builds a prompt containing:

- The dotted module name + file path of the failing module
- The full stderr traceback from the import attempt
- The source of the failing module (capped at 400 lines to keep the prompt bounded)
- The source of the related file if identified (also capped)
- Instructions to diagnose the root cause and propose a **specific** fix (install a package / edit file X line Y / rename a symbol / etc.) rather than generic advice

The LLM is called via `invoke_local_llm` with the model resolved from `LLM_AIDER_MODEL` (chains to `LLM_DEFAULT_MODEL` when blank). The returned diagnosis is written to the log under a `### Diagnosis` subsection for that failure. Any `LLMError` during diagnosis (timeout, empty response) is caught and logged; the overall run continues to the next failure rather than aborting.

### Single-Pass Design

Unlike the earlier iterative-repair version, this tool:

- Walks the package once
- Imports each module once
- For each failure, runs one LLM diagnosis
- Writes one log file
- Exits

No re-check, no retry, no cascading. If a proposed fix has downstream consequences, the user re-runs after applying.

### Logging

The log file is written progressively:

- Header (timestamps, package, model, endpoint) ‚Äî first
- `## Summary` (one-line-per-failure) ‚Äî written before any LLM calls, so an interrupted run still has the full failure list
- `## Full errors` with tracebacks + related-file paths ‚Äî written next
- `## Diagnoses` (one per failure) ‚Äî appended as each LLM call completes

If the user Ctrl+C's during diagnosis, all partial diagnoses already received are on disk.

## Exit Codes

| Code  | Meaning                                                                                                           |
| ----- | ----------------------------------------------------------------------------------------------------------------- |
| `0`   | All modules imported cleanly. No log entries beyond the header + "All modules import cleanly." line.               |
| `1`   | One or more modules failed to import. The log file contains the failures + diagnoses for manual review.           |
| `1` (via `sys.exit`) | `--package` directory not found.                                                                                  |

## Troubleshooting

| Symptom                                         | Cause                                       | Fix                                                                                                |
| ----------------------------------------------- | ------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| "Package directory not found"                   | Wrong `--package` path                      | Verify relative to cwd. Use `--package src/myapp`, not `--package myapp`.                          |
| All modules FAIL with `ModuleNotFoundError`     | Package not installed in editable mode      | Run `pip install -e .` in the project root first.                                                  |
| Import test passes locally but fails here       | Different Python interpreter                | Use `--python` to point at the correct venv.                                                       |
| Diagnosis takes a long time                     | 30B model + large package                   | Use `--no-diagnose` for the quick list first; diagnose selectively afterwards with a smaller model by setting `LLM_AIDER_MODEL` in `.env`.         |
| "Empty response from LLM" errors in the log     | Local model ran out of `num_predict` budget | Raise `LLM_FIX_MAX_TOKENS` in `.env` or switch to a non-thinking coder model.                      |
